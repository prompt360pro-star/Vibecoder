## 1. PREPARAÇÃO — Organizar Tudo para a IA

```
O PROBLEMA:
A IA tem LIMITE DE CONTEXTO. Você não pode colar
tudo de uma vez. Precisa de uma estratégia de
ALIMENTAÇÃO PROGRESSIVA.

A SOLUÇÃO:
Transformar toda a documentação em BLOCOS MODULARES
que a IA consome na ordem certa, no momento certo.
```

### 1.2 — Crie o MASTER-BRIEF.md

```
Este é O DOCUMENTO que você vai dar à IA SEMPRE no início
de qualquer sessão. É o "DNA" do projeto em 1 página.
```

```markdown
# VIBECODE — Master Brief

## O que é
App mobile (React Native/Expo) que ensina vibe coding
(programar com IA) do zero ao avançado, com gamificação
inspirada no Duolingo e mentor IA personalizado (Vi).

## Stack
- Mobile: Expo + React Native + Expo Router
- Backend: Next.js 14 API Routes
- Database: PostgreSQL (Neon) + Prisma
- Auth: Clerk
- AI: Claude (Anthropic) + GPT-4o (OpenAI) + Groq
- Cache: Redis (Upstash)
- State: Zustand + React Query
- UI: Custom components + Expo LinearGradient
- Deploy: Vercel (API) + EAS (mobile)

## Monorepo
- apps/mobile — Expo app
- apps/web — Next.js API
- packages/db — Prisma schema
- packages/shared — Types, schemas Zod, constants
- packages/ai — Prompts do Vi, model router

## Tema Visual
- Dark mode padrão
- Cores: bg #0A0A0F, card #1A1A2E, accent #8B5CF6
- Font: Inter (UI) + JetBrains Mono (code)
- Mascote: Vi (robô que evolui com o aluno)

## Funcionalidades Core (MVP)
1. Onboarding + DNA Test
2. Mapa de ilhas (4 níveis)
3. Missões interativas (30 no MVP)
4. Chat com Vi (mentor IA)
5. Sistema de XP, níveis e streaks
6. Perfil com badges e progresso
7. Sandbox de código
8. Auth (Clerk)

## Arquitetura de Prompts do Vi
- Base prompt (personalidade)
- Level adapter (6 níveis)
- Mode prompt (8 modos)
- Context layer (missão/projeto)
- Memory layer (memórias do aluno)
- Safety layer (guardrails)
- Montagem server-side apenas

## Database Models Principais
User, MissionProgress, ViConversation, ViMemory,
UserAchievement, StreakDay, Project, SocialPost,
Certificate, Notification
```

## 3. O PLANO DE CONSTRUÇÃO — 8 Sprints

```
REGRA DE OURO:
Construa em CAMADAS, não em features isoladas.
Cada sprint entrega algo FUNCIONAL que você pode
testar no celular real.

Sprint 1 → App abre e autentica
Sprint 2 → Home mostra dados reais
Sprint 3 → Uma missão funciona
Sprint 4 → Vi responde
Sprint 5 → Gamificação funciona
Sprint 6 → Perfil completo
Sprint 7 → Social básico
Sprint 8 → Polish + Deploy
```

## 4. REGRAS DE OURO PARA PEDIR CÓDIGO À IA

### 4.1 — Regras do Cursor

```
CRIE ARQUIVO: .cursorrules (na raiz do projeto)
```

```markdown
# VibeCode — Cursor Rules

## Projeto
App mobile de educação em vibe coding. Monorepo com Expo + Next.js.

## Stack
- Mobile: React Native + Expo SDK 51+ + Expo Router
- Backend: Next.js 14 App Router
- Database: PostgreSQL + Prisma
- Auth: Clerk
- AI: Anthropic Claude + OpenAI GPT + Groq
- State: Zustand + React Query
- Style: StyleSheet.create (não usar Tailwind no mobile)

## Convenções de Código
- TypeScript strict em todo lugar
- Functional components com arrow functions
- Named exports (nunca default, exceto pages do Next.js/Expo)
- Variáveis e funções em inglês
- Comentários em português
- 2 espaços de indentação
- Aspas simples para strings (exceto JSX que usa duplas)
- const > let, nunca var
- async/await > .then()
- Zod para validação em todos os endpoints
- Erro handling com try/catch em todo endpoint

## Padrões Mobile
- Cores do packages/shared/constants (COLORS)
- StyleSheet.create no final do arquivo
- Haptic feedback em todos os botões (expo-haptics)
- SafeAreaView quando necessário
- KeyboardAvoidingView em telas com input
- Platform.OS para código específico iOS/Android

## Padrões API
- Sempre autenticar com Clerk (auth())
- Sempre validar body com Zod
- Sempre retornar { success: boolean, data?, error? }
- Rate limit em endpoints sensíveis
- Log errors com console.error

## Estrutura de Arquivos
- Componentes: apps/mobile/components/[categoria]/[nome].tsx
- Hooks: apps/mobile/hooks/use-[nome].ts
- Services: apps/mobile/services/[nome].ts
- Stores: apps/mobile/stores/[nome]-store.ts
- API Routes: apps/web/app/api/[recurso]/route.ts

## O que NÃO fazer
- Não usar any (usar unknown se necessário)
- Não usar require (usar import)
- Não hardcodar cores (usar COLORS constant)
- Não hardcodar strings de UI (preparar para i18n)
- Não colocar lógica de negócio nos componentes
- Não fazer chamadas API diretas (usar hooks com React Query)
```
